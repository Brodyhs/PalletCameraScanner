### 1. Unguarded OSError in `EvidenceWriter.write_burst` silently eats a MissEvent — pending miss popped before emit, swallowed as a frame error

- **severity:** critical (verified 2–0, executed repro, 4 independent finders converged here)
- **file:** `palletscan/events/evidence.py`
- **line:** 79 (and 51)
- **summary:** `write_burst` still has two unguarded raisers on the miss path: `target.mkdir(parents=True, exist_ok=True)` at evidence.py:51 and `(target / 'meta.json').write_text(...)` at evidence.py:79. The module deliberately guards `cv2.imwrite` ("full disk... returns False, not raising", evidence.py:56-63) and hardened `prune()`/`_candidate_dirs` after REVIEW_7e4c22c finding #2 — but missed these two. An ENOSPC/PermissionError there propagates `write_burst` → `_finalize_miss`, where the `_PendingMiss` was already destructively popped (pass_tracker.py:174; 184-185 in flush) before the emit at pass_tracker.py:277. `_pipeline_loop` swallows it as `frame_errors += 1` and continues (app.py:362-366), so the MissEvent reaches no sink ever — even a one-shot transient fault loses it permanently because the pending entry is gone (reproduced: restoring permissions and flushing recovers nothing).
- **failure_scenario:** disk fills on the 24/7 station. Every undecoded pallet from then on is eaten one per finalize while decoded passes keep flowing: neither a stored PassEvent nor a stored MissEvent, exit code 0, dashboards healthy — per-camera passes-seen deflates and A/B read rate silently inflates; manifest reconciliation shows the pallets as never-seen with no evidence to review. At shutdown the same raise aborts the flush drain loop (via app.py:376-382), discarding all remaining pending misses at once (that path at least exits nonzero). Reproduced in four cases (mkdir raiser, meta.json raiser with imwrite failing soft as designed, healthy control, shutdown drain). Fix shape: emit the MissEvent (with `evidence_frame_count=0` / null `evidence_dir`) even when the evidence write fails, or degrade `write_burst` on OSError like its own prune helpers already do.

### 2. Motion segment spans a watchdog reconnect unbroken; a decoded pallet on the far side silently swallows the undecoded pallet's MissEvent

- **severity:** critical (both verifiers raised from high; verified 2–0, executed repro)
- **file:** `palletscan/reliability/watchdog.py`
- **line:** 146
- **summary:** after a reopen, `WatchdogSource.frames()` resumes yielding (watchdog.py:146) with no discontinuity signal, and CameraSource deliberately keeps ts monotonic across reconnects (camera.py:107, never re-anchored). `MotionGate` closes a segment only after `quiet_frames` *observed* inactive frames (motion_gate.py:103-105) — it never consults ts — so a segment open when the source stalled stays open across an arbitrarily long outage and glues onto whatever motion is present at reconnect. PassTracker's miss decision is per-segment: if ANY payload in the merged segment reaches confirmations, the segment is not a miss (pass_tracker.py:195-213).
- **failure_scenario:** camA opens a segment for pallet P1 whose label never decodes (the canonical miss case); the camera stalls mid-pass; the watchdog recovers in ~5-15 s while a second forklift (P2) is mid-zone, so the still-open segment continues; P2 decodes; the segment closes after P2 leaves. Reproduced with the real MotionGate+PassTracker: same segment id across a 12 s ts jump, `passes_emitted=1` (P2), `misses_emitted=0` — P1 evaporates and its reservoir evidence is discarded; P2's time-to-first-decode comes out as 12.33 s (outage-inflated A/B metric). The operator finds P1 only weeks later as manifest expected-but-not-scanned, with no MissEvent and no evidence burst to review.

### 3. `_luma_channel` fixed from the *configured* fourcc at construction; the warn-only run path then extracts the chroma plane when the device silently lands on a different packed format — a blind camera that emits neither passes nor misses

- **severity:** high (both verifiers raised from medium; verified 2–0, executed repro)
- **file:** `palletscan/sources/camera.py`
- **line:** 114
- **summary:** line 114 fixes the packed-YUV luma channel once from `cfg.fourcc` (UYVY → channel 1, else 0) and `frames()` applies it to every HxWx2 frame (line 272). ### 6. Supervisor death orphans the writer child; the replacement supervisor can never adopt or stop it, and the stop-file channel reports a successful stop while the orphan keeps scanning

- **severity:** high (verified 2–0, executed repro)
- **file:** `palletscan/reliability/supervisor.py`
- **line:** 248
- **summary:** nothing ties the child's lifetime to the supervisor's (no job object on Windows, no PDEATHSIG/process-group kill on POSIX), and the deploy design explicitly anticipates supervisor death (`install_service.ps1 -RestartCount 10`). When the supervisor dies while the child lives, the orphan keeps `palletscan.lock`; the Task-Scheduler-restarted supervisor spawns children that exit 4 forever (backoff to the 300 s cap), and a stop request during this state is consumed by the *spinning* supervisor (supervisor.py:248-251 or `_wait_for_exit_or_stop`, which only signals ITS child), which unlinks `supervisor.stop` and exits 0 — `stop_palletscan.ps1:40-50` takes file consumption as confirmation and prints "PalletScan stopped gracefully" while the orphan writer is still running.

Operationally: the operator stops "gracefully", edits the config, restarts — the OLD pipeline with the OLD config keeps running the A/B trial indefinitely; RUNBOOK §10 archival proceeds against a live writer; and when the orphan finally dies, up to 300 s of capture outage passes before the spinning supervisor's next retry wins the lock — pallets in that window produce neither PassEvent nor MissEvent.

### 11. `EventBus.shutdown` gives up after a 30 s join and callers proceed as if drained — queued events silently lost at exit 0; in station mode the business SENTINEL can overtake a still-draining per-camera bus

- **severity:** medium (confirmed 2–1 on adjudication; executed repros on all three verdicts)
- **file:** `palletscan/events/bus.py`
- **line:** 42
- **summary:** `shutdown()` puts the SENTINEL then `join(timeout=30)` and only logs on timeout (bus.py:42-44). `PipelineRunner.run` treats the return as fully drained (app.py:419), the CLI prints the summary and returns 0, and the daemon bus thread is killed at interpreter exit mid-drain: everything still in the 1024-deep queue (already-decoded passes and finalized misses) is dropped with exit 0 and no counter — contradicting the bus docstring ("never drops") and the RUNBOOK's flush guarantee. In station mode, `business_bus.shutdown()` (station.py:219) enqueues its SENTINEL while a straggling per-camera bus is still submitting through the deduper, so business events published after the SENTINEL are never handled, while `deduper.stats()` counters keep incrementing — the printed summary disagrees with the stored record.
- **failure_scenario:** scoping from the adjudication: under a supervisor-initiated stop the 15 s grace kill preempts the 30 s join (that loss path is documented and logged), so the silent-exit-0 variant lives on the child's *self-exit* paths — watchdog escalation (exit 3), pipeline failure, clean stop of a finite run — where `run()`'s finally executes `bus.shutdown()` with nothing racing it, plus any manual/foreground run. The trigger is one hung sink write: JsonlSink's write+flush is unbounded blocking I/O (sinks.py:65-70; SQLite's busy_timeout bounds lock waits, not a hung fsync), the canonical failing-eMMC/SD regime for a 24/7 box — and sink backpressure is itself a plausible escalation trigger, so the queue is non-empty precisely then. The tail of the shift's events ends in no sink; `events_handled` is quietly lower than passes+misses in the printed summary, exit 0.

### 14. Lossy `errors="replace"` payload decoding silently defeats manifest reconciliation for non-ASCII labels (Data Matrix path)

- **severity:** medium (verified 2–0, executed repros; gap-sweep finding)
- **file:** `palletscan/pipeline/decoders.py`
- **line:** 91 (verifier-corrected locus; the finder cited :60)
- **summary:** the pylibdmtx wrapper converts raw symbol bytes with `r.data.decode("utf-8", errors="replace")` (decoders.py:91), irreversibly replacing non-UTF-8 bytes with U+FFFD before the payload becomes the tracker dedup key (pass_tracker.py:216), the cross-camera key (dedup.py:134), and the manifest match key (manifest.py:66-81). The manifest *input* path was deliberately made strict-UTF-8 after the prior review (web/app.py:222-230 rejects non-UTF-8 uploads; web/store.py:156's own comment says U+FFFD payloads "can never match a scan") — yet the scan side keeps producing exactly those payloads, and the raw bytes are discarded at this line so nothing downstream can repair it. ISO/IEC 16022's *default* byte interpretation is Latin-1, so a fully spec-compliant DM label printer emitting any non-ASCII character produces this without misconfiguration. (Verifier corrections: the QR path corrupts differently — zbar transcodes byte-mode data upstream of :60, yielding valid-but-wrong UTF-8, no U+FFFD; and the distinct-pallets-collapse variant is real but narrow — see refuted #13.)
- **failure_scenario:** labels carry `PLT-MÜNCHEN-0042` in Latin-1. The camera decodes perfectly; the stored payload is `PLT-M�NCHEN-0042`. Invariant 1 holds (the PassEvent stores fine), but reconciliation buckets the expected payload as *missing* and the scan as *unexpected*, permanently deflating `true_read_rate` — the headline trial number reports a decoded pallet as never scanned. Environmental precondition: fires only if the site's labels contain non-ASCII bytes, and it leaves a forensic trail (U+FFFD glyphs adjacent in the missing/unexpected buckets) — hence medium, not high.

### 15. Supervisor startup unconditionally unlinks `supervisor.stop` as "stale", eating a concurrent live stop request — `stop_palletscan.ps1` reports "stopped gracefully" while the station starts up

- **severity:** medium (verified 2–0, executed repro)
- **file:** `palletscan/reliability/supervisor.py`
- **line:** 206
- **summary:** `Supervisor.run()` deletes any existing stop-file at startup before spawning the child (supervisor.py:206-210), with no age/ownership check. `stop_palletscan.ps1:36-50` writes the file and treats its disappearance as confirmation of a graceful stop. The protocol cannot distinguish "supervisor consumed the request and exited" from "a (re)starting supervisor discarded the request and is now running": a stop written while a supervisor is booting — e.g. during Task Scheduler's 1-minute auto-restart window after a supervisor failure — is unlinked as stale, the child spawns, and the stop script prints success.

The operator, seeing the station down, runs the stop script at T+59 s for maintenance: the restarting supervisor unlinks the request at T+60.5 s and spawns the pipeline; the script (polling at 500 ms) sees the file vanish and prints "PalletScan stopped gracefully". Cameras get unplugged and `palletscan.db` archived while the station is actually scanning — passes during cable handling are recorded as misses or hardware outages, and the archival move races a live SQLite writer.

## Refuted during verification (recorded so they aren't re-raised)

All refuted with executed probes except where noted.

1. **Pending misses frozen in RAM during an outage (`pass_tracker.py:173`):** premise true, but every claimed loss path fails — the first recovered frame finalizes all pending misses at once (monotonic ts across reopen is the deliberate hook), and every stop path funnels through the pipeline-thread finally flush. The only loss is a no-signal hard kill, the documented in-flight boundary.
2. **Supervisor 15 s grace < child's ~45 s drain budget (`supervisor.py:181`):** the arithmetic sums phases that occur *after* events are already durable — the pipeline thread's own finally flushes evidence and publishes events before the executor/bus joins begin. Probe with a 20 s-wedged executor and dead HTTP endpoint lost nothing.
3. **Evidence burst + prune stalls past the 64-frame queue horizon (`evidence.py:82`):** burst is hard-bounded at ~106 JPEGs by the reservoir cap + post-roll; the measured full worst case (1080p, at-cap tree) is 0.77 s vs the 2.13 s horizon; the at-cap prune scan is 40 ms. Zero drops.
4. **HTTP re-emits share an event_id, so receivers deduping on event_id lose merges (`http_sink.py:12`):** deliberate, documented, owner-approved (ASSUMPTIONS #40 "Owner amendment — re-emit ordering"); A/B numbers come exclusively from the local revision-guarded store.
5. **No atomicity between SQLite commit and outbox enqueue (`http_sink.py:124`):** mechanically accurate dual-write gap, but the invariant-bearing local store is durable first; the loss is one row in the best-effort remote mirror after a hard kill timed into a sub-millisecond window — the documented hard-stop boundary.
6. **Supervisor gives up permanently on a transient spawn OSError (`supervisor.py:218`):** the "any failure restarts" policy is about child *exit codes*; a spawn failure has no child, exits 1 loudly, and Task Scheduler restarting the supervisor is the explicit, owner-approved recovery layer (PLAN_PHASE5: "PT1M is fine for that rare case").
7. **VideoFileSource treats mid-file decode failure as clean EOF (`video.py:109`):** empirically false for the shipped MJPG/.avi format — a corrupt frame is skipped (99/100 delivered), not EOF'd; and a power-cut AVI reports `CAP_PROP_FRAME_COUNT` 0.0, so the proposed expected-count cross-check cannot exist.
8. **Replay-mode soak performs zero event accounting (`tools/soak.py:290`):** documented tool scope, not a bug — replay soak is the memory-flatness measurement (RUNBOOK/README/docstring all say so) and `--inject-every-s` hard-errors under replay "(needs ground truth)"; the no-event-loss proofs belong to synthetic injection mode.
9. **Soak unlinks `outbox.db` in whatever `--data-dir` it is given (`tools/soak.py:335`):** requires explicitly overriding the safe harness-owned default `data/soak`; every documented invocation omits `--data-dir`, and config rebasing keeps the unlink inside the soak dir.
10. **Cap fields accept zero/negative values (`config.py:209`):** validation-hardening suggestion, not a correctness bug — a 0 MB cap producing maximal pruning is the documented cap semantics, and no shipped "0 = unlimited" convention exists for caps (each 0-special-case is individually commented elsewhere).
11. **An undecoded pallet sharing a motion segment with a decoded one vanishes (`pass_tracker.py:200`):** spec-conformant by the project's own definition — Prompt.md §4 defines the 100% mechanism at motion-pass granularity, pass_tracker's docstring states the same rule, and ASSUMPTIONS #13 explicitly anticipates multi-pallet segments. The motion candidate is always accounted.
12. **`synth`/`replay` lack the RuntimeError→exit-code mapping `run` has (`cli.py:770`):** the asymmetry exists but the operational claim fails — both paths still exit 1 with the identical `restarts.jsonl` record, and WatchdogEscalation (the only exit-3 case) is unreachable for replay/synth sources.
13. **`errors='replace'` collapses *distinct* non-UTF-8 payloads into one pallet (`decoders.py:60`):** the finder's literal pair doesn't collide (replacement preserves the differing trailing bytes), the QR path is immune (zbar transcodes upstream), and ASSUMPTIONS #8's deployed label scheme is ASCII. The reachable half of this site — manifest reconciliation mismatch on the DM path — is confirmed finding 14.
